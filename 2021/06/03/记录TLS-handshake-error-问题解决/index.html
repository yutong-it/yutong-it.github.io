记录TLS handshake error 问题解决

发表于 2021-06-03

记录一次问题排查过程

有同事反馈说k8s的api日志里一直有报错，刚开始也是没有头绪，后来找到了原因，现总结下排查过程
当然这次排查问题时间比较久是因为基础知识缺乏，比如对keepalived和haproxy的功能和配置不熟悉导致
先说问题吧
k8s的api日志里有如下报错

看到第一眼觉得应该是https的问题，在握手的时候失败了
没有其他有用信息了，网上相关的信息也很少，但还是找到了一个有价值的博客
在此，感谢那位博主，” 落魄运维” ，博客地址： https://www.cnblogs.com/Dev0ps/p/12102388.html
博主的也遇到了类似的问题，博主是这样解决的

zabbix一直监控api的端口，导致此问题，但是我这边并没有zabbix，但是
我的问题应该和他的类似，都是某个程序一直在访问我api，而且是http的那种，并不是https，所以一直在报错
好的，我的目标有了，就是找到这个程序
1、万里长征第一步，抓包

因为api监听的端口在6443,所以我抓取6443上的tcp包，根据日志里EOF前面的端口号进行过滤
理论上如果有连接就应该能抓到包，可惜我过滤了好多次，都没找到，这条路放弃了
2、第二步，这个日志应该出现的时候有最开始的时间，是不是有同事在那个时候启动了
什么程序，导致访问api出错，所以我开始查看所有在落在本台服务器上的pod的启动日期

查看了几个pod，并没有发现什么异常
3、我仔细观察了下日志，发现他是有规律的，每次间隔都是6秒，所以他不是大量连接超时导致
应该也是监控导致，那谁在监控这个端口呢，由于集群采用keepalived+haproxy的高可用方式
所以问题应该是出在这两个程序上，但是对这两个并不了解，只是知道他们是提供了一个api的
vip可以访问
先查下keepalived的吧
keepalived提供了一个虚ip，比如有3个master节点，每个节点启动keepalived程序
3个master之间会选出一个主，2个为备，提供这个虚拟ip的流量转发
haproxy
haproxy提供了对apiserver的反向代理功能，他将接受的请求轮询转发到后端的apiserver

了解了以上，就可以知道haproxy会监控后端server的存活，赶紧去看haproxy的配置

难道是haproxy的检查导致的
为了验证，手动去telnet api server
echo “” |telnet 172.28.0.11 6443
发现请求多次并没有EOF的日志出现，可能不是hproxy的原因
去查看keepalived的配置

发现有个chk_proxy，查了下这个就是监控，所以非常有可能是此导致的，而且网上的配置一般是
”killall -0 haproxy“
替换#script “nc -w 3 -z 127.0.0.1 4443”
